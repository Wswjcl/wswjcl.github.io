RAG 系列（六）：重排序与上下文整合 — 最后的精炼

发表于2026-07-10|更新于2026-07-13|技术
|浏览量:

检索返回了 Top-K 个候选文档块，但它们的排序未必最优——向量相似度只是一个粗略的相关性信号。重排序（Reranking）对检索结果做精细化排序，而上下文整合则决定如何将这些文档组织为 LLM 的输入。这两个环节是检索与生成之间的”精炼器”，直接影响最终回答的质量。

一、为什么需要重排序？

1.1 检索排序的局限

向量检索的相似度得分存在系统性偏差：

  • 语义漂移：嵌入是整段文本的”平均”表示，可能被文本中的次要话题稀释
  • 长度偏差：较长的文本块因包含更多词汇，可能与更多查询有非零相似度
  • 流行度偏差：某些高频主题的文档可能对多种查询都有中等相似度
  • 查询-文档不对称：查询通常很短，文档很长，两者的语义空间分布不同

1.2 双编码器 vs 交叉编码器

检索阶段使用的嵌入模型通常是双编码器（Bi-Encoder）：查询和文档分别编码，再计算向量相似度。这种方式高效（文档向量可预计算），但查询和文档之间没有交互——编码时看不到对方。

交叉编码器（Cross-Encoder） 将查询和文档拼接后一起送入模型，允许两者在所有 Transformer 层中充分交互：

交叉编码器的优势：

  • 精细交互：查询的每个 token 都能 attend 到文档的每个 token
  • 更高精度：在重排序任务上显著优于双编码器

交叉编码器的劣势：

  • 无法预计算：每对 (查询, 文档) 都需要一次前向传播
  • 计算成本高：对 $K$ 个候选文档需要 $K$ 次模型推理

这正是 RAG 的经典分工：双编码器做粗筛（检索），交叉编码器做精排（重排序）。

2.1 交叉编码器 Reranker

Cohere Rerank

Cohere 提供的 API 重排序服务：

  • 无需自托管，API 调用即可
  • 支持多语言
  • 效果优秀，MTEB Reranking 排名靠前
  • 按调用次数计费

ColBERT Reranker

基于 ColBERT 的延迟交互架构，精度接近交叉编码器，但可以利用预计算的文档 token 嵌入加速：

$$\text{score}(q, d) = \sum_{i=1}^{|q|} \max_{j=1}^{|d|} \text{sim}(q_i, d_j)$$

三、重排序流程

3.1 分数校准

不同 Reranker 的分数量级不同，级联使用时需要校准：

  • 归一化：将分数映射到 $[0, 1]$
  • 温度缩放：$\text{score}’ = \text{softmax}(\text{score} / T)$，$T$ 控制分布锐度
  • 阈值过滤：低于阈值的文档直接丢弃，无论排名如何

3.2 多样性控制

重排序后可能存在冗余：多个高排名文档内容高度相似。可以加入多样性惩罚：

  • MMR（Maximal Marginal Relevance）：在相关性和多样性之间权衡

$$\text{MMR}(d) = \lambda \cdot \text{Rel}(d) - (1 - \lambda) \cdot \max_{d’ \in S} \text{Sim}(d, d’)$$

其中 $S$ 是已选文档集，$\lambda$ 控制相关性与多样性的权衡。

四、上下文窗口管理

重排序后，需要将选定的文档块组织为 LLM 的输入上下文。这一步看似简单，实则暗藏玄机。

4.1 Lost in the Middle

研究表明，LLM 对上下文不同位置的信息利用率不均匀——开头和结尾的信息被更好地利用，中间位置的信息容易被”忽略”：

这一现象被称为 Lost in the Middle（Liu et al., 2023）。

4.2 上下文组织策略

顺序策略

策略 排列方式 优点 缺点
按相关性降序 最相关的放最前 LLM 优先看到最佳信息 次相关信息被放在中间可能被忽略
按相关性升序 最相关的放最后 利用 LLM 对末尾的注意力 开头放低质量信息浪费 token
两头放、中间弱 最相关放首尾，次相关放中间 规避 Lost in the Middle 需要精心安排

推荐策略：相关性降序 + 关键信息首尾

将最相关和次相关的文档分别放在开头和结尾，确保关键信息不被遗漏。

4.3 上下文压缩

当检索结果超出 LLM 的上下文窗口时，需要压缩：

  • 截断：直接截掉低排名文档（最简单但粗暴）
  • 提取式压缩：从每个文档中提取与查询最相关的句子
  • 生成式压缩：用 LLM 对每个文档生成与查询相关的摘要
  • 上下文蒸馏：用小模型将长文档压缩为短表示

五、Prompt 工程与上下文组织

5.1 RAG Prompt 模板

一个良好的 RAG Prompt 应包含：

  1. 角色定义：告诉 LLM 它的角色和行为规范
  2. 上下文注入：将检索到的文档注入
  3. 引用要求：要求 LLM 基于上下文回答并标注来源
  4. 不确定性处理：告诉 LLM 上下文不足时如何应对

六、小结

重排序与上下文整合是检索与生成之间的”精炼器”，决定了 LLM 看到什么、以什么顺序看到。本文的核心要点：

  • 交叉编码器通过查询-文档全交互实现精排，是 Reranker 的主流选择
  • LLM-based Reranker 精度更高但成本更高，适合对 Top-K 做最终精排
  • Lost in the Middle 问题要求精心安排上下文顺序，关键信息放首尾
  • Prompt 工程是上下文整合的重要一环，好的 Prompt 能显著提升回答质量
  • 多样性控制（MMR）和上下文压缩避免冗余信息浪费 token
