RAG 系列（七）：评估与优化 — 闭环提升

发表于2026-07-10|更新于2026-07-13|技术
|浏览量:

RAG 系统搭建完成后，如何知道它好不好？哪里需要改进？评估是 RAG 工程化落地的关键一环——没有评估，优化就是盲人摸象。同时，RAG 技术本身也在快速演进，GraphRAG、Self-RAG、Adaptive RAG 等新范式不断涌现。

本文将系统讲解 RAG 的评估方法与前沿趋势，为整个系列收尾。

一、为什么评估如此重要？

RAG 是一个多组件级联系统，任何一个环节的短板都会影响最终效果。评估的核心价值在于：

  1. 定位瓶颈：是检索召回不够？还是重排序不准？还是 LLM 没用好上下文？
  2. 指导优化：知道哪里差，才能有针对性地改进
  3. 回归保障：修改某环节后，确保不会导致其他指标退化
  4. 效果量化：向利益相关方证明系统的实际价值

评估 → 优化 → 再评估，形成闭环。

二、RAG 评估框架

2.1 RAGAS（Retrieval Augmented Generation Assessment）

RAGAS 是目前最流行的 RAG 评估框架，由 Es et al. 于 2023 年提出，提供了一套无需人工标注即可评估 RAG 的指标体系。

核心指标

指标 评估对象 含义 计算方式
Faithfulness 生成 回答是否忠实于检索到的上下文 将回答拆分为原子声明，检查每条声明是否可由上下文推出
Answer Relevancy 生成 回答是否与问题相关 用 LLM 从回答反推问题，计算与原问题的语义相似度
Context Precision 检索 检索结果中相关文档的排名是否靠前 逐位检查：相关文档是否排在不相关文档前面
Context Recall 检索 检索结果是否覆盖了回答所需的所有信息 将标准答案拆分为原子声明，检查每条声明是否可由上下文推出

Faithfulness 详解

$$\text{Faithfulness} = \frac{|\text{可由上下文推出的声明数}|}{|\text{总声明数}|}$$

如果回答中包含上下文没有的信息（如”由 Google 提出”），且上下文确实没提到 Google，则 Faithfulness < 1.0。

Context Recall 详解

$$\text{Context Recall} = \frac{|\text{可由上下文推出的答案声明数}|}{|\text{总答案声明数}|}$$

Context Recall 衡量检索的信息覆盖度——如果标准答案中的关键信息在检索结果中找不到，说明检索有遗漏。

2.2 TruLens

TruLens 是另一个流行的 RAG 评估框架，提供实时监控和评估能力：

指标 含义
Groundedness 回答的每个声明是否基于检索上下文（类似 Faithfulness）
Answer Relevance 回答是否与问题相关
Context Relevance 检索到的上下文是否与问题相关

TruLens 的特色是实时仪表盘，可以在生产环境中持续监控 RAG 的质量。

2.3 DeepEval

DeepEval 提供了更丰富的评估指标和测试框架：

  • Hallucination：检测回答中的幻觉内容
  • Bias：检测回答中的偏见
  • Toxicity：检测回答中的有害内容
  • Conversational：评估多轮对话的连贯性

三、组件级评估

3.1 检索评估

检索是 RAG 的上游，其质量决定了整个系统的上限。

指标 公式 含义
Recall@K $\frac{ \text{Top-K} \cap \text{relevant}
Precision@K $\frac{ \text{Top-K} \cap \text{relevant}
MRR $\frac{1}{|Q|} \sum_{i=1}^{|Q|} \frac{1}{\text{rank}_i}$ 第一个相关文档的排名倒数均值
nDCG@K $\frac{\text{DCG@K}}{\text{IDCG@K}}$ 考虑排序位置的归一化增益
MAP $\frac{1}{|Q|} \sum_{i=1}^{|Q|} \text{AP}_i$ 平均精度均值

其中 DCG 定义为：

$$\text{DCG@K} = \sum_{i=1}^{K} \frac{2^{\text{rel}_i} - 1}{\log_2(i + 1)}$$

实践建议：RAG 场景中最关注 Recall@K——只要相关信息被检索到，后续的重排序和生成有机会把它用好。

3.2 重排序评估

重排序的评估关注排序质量：

  • nDCG：最常用，衡量排序与理想排序的接近程度
  • MAP：衡量相关文档是否排在前面
  • Precision@K：前 K 个中有多少是相关的

3.3 生成评估

生成的评估关注回答本身的质量：

指标 评估方式
Faithfulness LLM-as-Judge：让 LLM 判断回答是否基于上下文
Answer Relevancy 语义相似度：回答与问题的相关程度
Correctness 与标准答案对比（需人工标注）
Completeness 回答是否覆盖了问题的所有方面
Conciseness 回答是否简洁，无冗余

四、端到端评估

4.1 人工评估

最可靠的评估方式，但成本高：

  • 正确性：回答是否事实正确
  • 完整性：回答是否充分
  • 流畅性：回答是否自然通顺
  • 有用性：回答是否解决了用户的问题

五、常见问题与优化方向

5.1 幻觉问题

症状：回答包含检索上下文中没有的信息
根因：LLM 的参数知识”泄漏”，或 Prompt 未充分约束
优化：

  • 强化 Prompt 约束：”仅基于以下参考资料回答”
  • 检查 Faithfulness 指标，定位幻觉来源
  • 增加重排序的严格度，过滤低相关度文档

5.2 检索失败

症状：正确信息未被检索到，Context Recall 低
根因：分块不当、嵌入模型不匹配、查询表述与文档不匹配
优化：

  • 优化分块策略（调整 chunk size、换用语义分块）
  • 尝试不同的嵌入模型
  • 加入查询改写（HyDE / Multi-Query）
  • 启用混合检索

5.3 冗余信息

症状：检索结果中大量无关内容，浪费上下文窗口
根因：检索 Top-K 过大、重排序不够严格
优化：

  • 降低 Top-K 或加入相关性阈值过滤
  • 加强重排序（使用更精准的 Reranker）
  • 加入上下文压缩
  • 使用 MMR 控制多样性

5.4 延迟过高

症状：端到端响应时间过长
根因：多环节级联、大模型推理慢
优化：

  • 缓存热门查询的结果
  • 使用更小的 Reranker 或 LLM
  • 流式输出（Streaming）降低首字延迟
  • 异步并行执行独立步骤

六、RAG 的前沿趋势

6.1 GraphRAG

微软提出的 GraphRAG 将知识图谱（Knowledge Graph）引入 RAG，解决传统 RAG 在全局性问题上的不足。

传统 RAG 的局限：擅长回答局部细节问题（”X 是什么？”），但不擅长回答需要综合全局信息的问题（”所有文档的主要主题是什么？”）。

GraphRAG 的方法：

  • 局部查询：用传统向量检索回答细节问题
  • 全局查询：用社区摘要回答需要综合全局信息的问题

论文：Edge et al., From Local to Global: A Graph RAG Approach to Query-Focused Summarization, 2024.

6.2 Self-RAG

Self-RAG 让 LLM 学会自我反思：在生成过程中自主决定是否需要检索、检索结果是否有用、生成的回答是否忠实。

Self-RAG 通过三种反思 token 实现：

  • Retrieve token：是否需要检索？
  • IsRel token：检索结果是否相关？
  • IsSup token：回答是否被上下文支持？

论文：Asai et al., Self-RAG: Learning to Retrieve, Generate, and Critique through Self-Reflection, ICLR 2024.

6.3 Adaptive RAG

Adaptive RAG 根据查询的复杂度自适应选择检索策略：

复杂度分类器可以是一个小模型（如 T5），训练数据标注为 {简单, 中等, 复杂} 三类。这种自适应策略在保证简单查询的响应速度的同时，对复杂查询投入更多计算资源。

6.4 CRAG（Corrective RAG）

CRAG 在检索后加入评估-纠正环节：

  1. 检索文档后，用轻量评估器判断检索质量
  2. 如果评估器认为检索结果可靠 → 正常使用
  3. 如果评估器认为检索结果不可靠 → 用 Web 搜索补充/替代

论文：Yan et al., Corrective Retrieval Augmented Generation, ICML 2024.

6.5 前沿趋势总结

范式 核心创新 解决的问题 成熟度
GraphRAG 知识图谱 + 社区摘要 全局性问题 ⚠️ 中
Self-RAG LLM 自我反思 不必要的检索 / 幻觉 ⚠️ 中
Adaptive RAG 查询复杂度自适应 简单查询过重 / 复杂查询过轻 ⚠️ 中
CRAG 检索评估 + 纠正 检索失败无补救 ⚠️ 中
Modular RAG 组件可插拔 架构僵化 ✅ 高

七、系列总结

本系列从 RAG 的概述出发，逐环节深入讲解了 RAG 管线的核心技术：

核心要点回顾

环节 关键要点
文档处理与分块 递归字符分块是通用起点；父子文档策略兼顾检索精度与上下文完整性
嵌入模型 对比学习 + 硬负样本是训练核心；中文推荐 BGE/GTE 系列
向量存储与索引 HNSW 是综合最优索引；量化技术是大规模检索的关键
检索策略 混合检索（稀疏+稠密）是标配；查询改写显著提升召回
重排序与上下文整合 交叉编码器精排；注意 Lost in the Middle，关键信息放首尾
评估与优化 RAGAS 指标体系；组件级评估定位瓶颈；GraphRAG/Self-RAG 是前沿方向

RAG 仍在快速演进中。从 Naive RAG 到 Advanced RAG，再到 Modular RAG 和各种新范式，核心趋势是：更智能的检索、更精准的排序、更可靠的生成、更闭环的评估。理解这些基础技术，是跟上 RAG 发展浪潮的根基。

Transformer 详解
一、背景与动机在 Transformer 出现之前，序列建模任务（机器翻译、文本生成等）的主流架构是 RNN 及其变体 LSTM / GRU。这些模型虽然在当时取得了不错的效果，但存在几个根本性的局限： 顺序计算瓶颈：RNN 必须逐步处理序列中的每个时间步，无法充分并行化。序列越长，训练越慢。 长距离依赖问题：尽管 LSTM 引入了门控机制来缓解梯度消失，但在实际中捕获超长距离的依赖关系仍然困难。 信息压缩瓶颈：RNN 将整个序列的信息压缩到一个固定维度的隐状态中，信息容易丢失。 2017 年，Vaswani 等人在论文 Attention Is All You Need 中提出了 Transformer 架构，完全摒弃了循环和卷积结构，仅依靠注意力机制（Attention）来建模序列间的依赖关系。这一设计带来了两大优势： 全并行计算：所有位置可以同时计算，极大提升了训练效率。 全局感受野：每个位置可以直接关注序列中任意其他位置，天然支持长距离依赖。 Transformer 的出现彻底改变了 NLP...
