RAG 系列（二）：文档处理与分块 — RAG 的基石

发表于2026-07-09|更新于2026-07-09|技术
|浏览量:

在 RAG 系统中，文档处理与分块是最上游的环节，却往往被低估。分块质量直接决定了检索的精度和生成的好坏——毕竟，如果送入检索的文本块本身就残缺或冗余，后续再精巧的检索与重排序也无济于事。正如那句老话：Garbage in, garbage out。

本文将系统性地讲解 RAG 中文档处理与分块的核心技术。

一、文档解析：从原始文件到纯文本

现实世界的知识以各种格式存在——PDF、Word、HTML、Markdown、PPT、表格、图片……文档解析的任务是将这些异构格式转换为结构化的纯文本，为后续分块和嵌入做准备。

1.1 常见文档格式的解析挑战

格式 挑战 常用工具
PDF 版面复杂、多栏、表格、公式、扫描件 PyMuPDF、pdfplumber、Unstructured、Marker
Word 样式信息丰富、嵌入对象 python-docx、Unstructured
HTML 标签噪声、导航栏与正文混合 BeautifulSoup、Readability、trafilatura
Markdown 结构相对规范，但代码块需特殊处理 直接读取 / markdown-it
PPT 文本碎片化、图文混排 python-pptx、Unstructured
表格 结构化数据，行列语义 Camelot、Tabula、pdfplumber
图片/扫描件 需要 OCR 识别 Tesseract、PaddleOCR、云服务 OCR

1.2 PDF 解析：最棘手的难题

PDF 是企业中最常见的文档格式，也是解析难度最高的：

  • 版面还原：PDF 中的文本流并不等同于阅读顺序。多栏排版、页眉页脚、脚注等都会干扰文本提取。
  • 表格识别：PDF 中的表格本质是定位指令，而非结构化数据，需要专门的表格检测与还原。
  • 公式与符号：数学公式在 PDF 中通常以字体/图形方式编码，难以准确还原为 LaTeX。
  • 扫描件：扫描的 PDF 需要先进行 OCR，而 OCR 本身就有识别误差。

1.3 解析质量的关键指标

  • 文本完整性：是否遗漏了原文中的关键信息？
  • 结构保留度：标题层级、列表、表格结构是否被正确还原？
  • 阅读顺序：提取的文本是否按照人类阅读顺序排列？
  • 噪声水平：是否混入了页眉页脚、水印、页码等无关内容？

二、文本分块：为什么需要分块？

分块（Chunking）是将长文档切分为较小文本片段的过程。为什么不能直接把整篇文档送入嵌入模型或 LLM？

  1. 嵌入模型的输入限制：嵌入模型（如 BGE、E5）通常有最大输入长度（512 或 8192 token），整篇文档远超此限制。
  2. 语义精度：整篇文档的嵌入是所有内容的”平均”，会模糊局部语义。一篇同时讨论 A 和 B 的文档，其嵌入无法精确匹配只关于 A 的查询。
  3. 检索粒度：检索时希望返回的是与查询最相关的片段，而非整篇文档，否则会引入大量无关信息。
  4. 上下文窗口：LLM 的上下文长度有限，检索结果需要在有限空间内提供最相关的信息。

三、分块策略详解

3.1 固定大小分块（Fixed-size Chunking）

最简单的策略：按照固定的字符数或 token 数切分，通常配合重叠（overlap）使用。

参数：

  • chunk_size：每个分块的大小（字符数或 token 数）
  • chunk_overlap：相邻分块的重叠量

优点：实现简单，分块大小均匀，易于控制
缺点：可能在句子或段落中间截断，破坏语义完整性

适用场景：格式统一的文本（如日志、合同条款），对语义完整性要求不高的场景

3.2 递归字符分块（Recursive Character Chunking）

LangChain 中最常用的分块策略。按优先级依次尝试不同的分隔符进行分割：

  1. 先按 \n\n（段落）分割
  2. 如果某段仍超长，再按 \n（换行）分割
  3. 如果仍超长，按句号等标点分割
  4. 最后才按字符数硬切

优点：尽可能保留语义边界，兼顾大小均匀
缺点：分块大小不完全一致，需要设置容差

适用场景：通用场景，是大多数 RAG 应用的默认选择

3.3 文档结构分块（Document Structure Chunking）

利用文档的天然结构（标题、章节、段落）进行分块：

层级策略：

  • 按最高级标题（H1）分块：粒度最粗，每个 chunk 是一整章
  • 按二级标题（H2）分块：粒度适中，每个 chunk 是一个小节
  • 按三级标题（H3）分块：粒度较细，每个 chunk 是一个子节

优点：完美保留文档的语义结构和层级关系
缺点：依赖文档有良好的标题层级；章节长度差异大，可能需要二次切分

适用场景：结构规范的文档（技术文档、论文、法律条文）

3.4 语义分块（Semantic Chunking）

基于语义相似度进行分块：计算相邻句子的嵌入向量相似度，当相似度低于阈值时切分。

阈值策略：

  • 百分位法：取所有相邻相似度的第 X 百分位作为阈值
  • 标准差法：取均值减去 K 倍标准差作为阈值
  • 插值法：在相似度骤降点切分

优点：语义边界最精确，分块内话题一致
缺点：计算成本高（需对所有句子做嵌入），速度慢；阈值选择敏感

适用场景：对检索精度要求极高的场景

3.5 Agentic 分块

利用 LLM 理解文档内容，自主决定如何分块。LLM 逐段阅读文档，判断语义转折点并切分。

优点：语义理解最准确，能处理隐式的话题转换
缺点：成本极高（每个文档都需要 LLM 调用），速度最慢
适用场景：高价值文档、对质量要求极致的场景

分块策略对比

策略 语义完整性 大小均匀性 计算成本 实现难度
固定大小 ❌ 低 ✅ 高 ✅ 低 ✅ 低
递归字符 ⚠️ 中 ⚠️ 中 ✅ 低 ✅ 低
文档结构 ✅ 高 ❌ 不均匀 ✅ 低 ⚠️ 中
语义分块 ✅ 高 ⚠️ 中 ❌ 高 ⚠️ 中
Agentic ✅ 最高 ❌ 不均匀 ❌ 最高 ❌ 高

四、Chunk Size 与 Overlap 的权衡

Chunk Size 是 RAG 系统中最关键的超参数之一，它直接影响检索和生成的效果。

4.1 小 Chunk 的利弊

优势：

  • 检索精度高：嵌入向量更聚焦于特定话题，匹配更精准
  • 上下文利用率高：同样的上下文窗口能放入更多独立信息片段
  • 适用于精确匹配型查询

劣势：

  • 语义不完整：可能切断上下文依赖，丢失关键信息
  • 需要更多分块：增加索引和检索的计算量
  • 缺乏全局上下文：只看到局部，可能误解含义

4.2 大 Chunk 的利弊

优势：

  • 语义完整：更容易保留完整的论证和上下文
  • 上下文丰富：单个 chunk 包含更多关联信息

劣势：

  • 检索噪声多：嵌入向量被稀释，匹配精度下降
  • 上下文窗口浪费：少量大 chunk 就可能占满 LLM 上下文
  • 可能包含大量无关信息

4.3 经验法则

Overlap 的作用：在相邻 chunk 之间保留重叠区域，确保跨 chunk 的信息不被截断。通常设置为 chunk size 的 10%-20%。

4.4 自适应分块大小

实际中，一刀切的 chunk size 并非最优。更先进的做法是：

  • 按段落自适应：以段落为最小单位，合并短段落、拆分长段落
  • 按内容类型调整：代码块、表格、列表使用不同的分块策略
  • 父子文档策略（Parent-Child）：小 chunk 用于检索，检索到后返回其所属的大 chunk 用于生成

这种策略兼顾了检索精度（小 chunk 嵌入更精准）和上下文完整性（大 chunk 包含完整信息），是 Advanced RAG 中的经典技巧。

五、元数据提取与保留

每个文本块除了内容本身，还应携带元数据（Metadata），这对后续的检索过滤和结果呈现至关重要。

5.1 常见元数据字段

字段 示例 用途
source 文档名 / URL 来源追溯
page 第 12 页 定位原文位置
section “3.2 模型架构” 结构化定位
author “Zhang et al.” 作者过滤
date 2024-03-15 时效性过滤
doc_type paper / manual / faq 文档类型过滤
keywords [“Transformer”, “Attention”] 辅助检索
chunk_index 5 / 20 块顺序信息

六、分块评估

如何评估分块策略的优劣？可以从以下几个维度考量：

6.1 内在质量指标

  • 语义完整性：分块内的文本是否构成语义完整的单元？
  • 信息冗余度：不同分块之间是否存在不必要的重复？
  • 大小分布：分块大小的方差是否合理？

6.2 下游任务指标

  • 检索召回率：正确答案所在的分块是否被检索到？
  • 检索精确率：检索到的分块中有多少是相关的？
  • 端到端回答质量：最终生成的回答是否准确、完整？

6.3 实用评估方法

最直接的评估方式是构建问答对测试集：

  1. 从文档中人工标注一批（问题, 答案, 来源段落）三元组
  2. 用不同的分块策略构建索引
  3. 对每个问题执行检索，检查正确段落是否在 Top-K 结果中
  4. 比较不同策略的召回率

七、实践建议

综合以上分析，给出 RAG 分块的实际建议：

  1. 起步：使用递归字符分块（chunk_size=512, overlap=50），简单且效果不差
  2. 优化方向：根据文档类型选择合适的策略——结构化文档用文档结构分块，非结构化文档用递归分块
  3. 进阶：采用父子文档策略，小 chunk 检索 + 大 chunk 生成
  4. 元数据：始终保留来源、页码、章节等元数据，支持过滤和追溯
  5. 实验驱动：用问答测试集评估不同分块策略的效果，用数据说话而非拍脑袋
  6. 迭代优化：分块不是一次性的，随着知识库和查询模式的变化持续调优

八、小结

文档处理与分块是 RAG 的基石，直接决定了知识库的质量上限。本文的核心要点：

  • 文档解析需要针对不同格式选择合适的工具，PDF 是最大挑战
  • 分块策略从简单到复杂：固定大小 → 递归字符 → 文档结构 → 语义分块 → Agentic
  • Chunk Size 的选择需要在检索精度和上下文完整性之间权衡
  • 父子文档策略是兼顾两者的有效方案
  • 元数据是常被忽视但至关重要的信息
