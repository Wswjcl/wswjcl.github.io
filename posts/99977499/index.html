RAG 系列（一）：概述 — 从检索到生成的范式革命

发表于2026-07-09|更新于2026-07-09|技术
|浏览量:

一、什么是 RAG？

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将信息检索与文本生成相结合的技术范式。其核心思想是：在大型语言模型（LLM）生成回答之前，先从外部知识库中检索相关信息，将检索到的内容作为上下文提供给 LLM，从而让模型的回答基于事实而非纯粹的参数记忆。

2020 年，Lewis 等人在论文 Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks 中首次系统性地提出了 RAG 框架，将预训练的检索器与序列到序列模型相结合，在开放域问答等任务上取得了显著提升。

二、为什么需要 RAG？

大语言模型虽然能力强大，但存在几个根本性的局限，RAG 正是为解决这些问题而生：

2.1 知识截止问题

LLM 的知识来自训练数据，存在明确的截止日期。对于训练之后发生的事件、最新发布的论文、实时变化的数据，模型一无所知。RAG 通过检索外部知识库，让模型能够访问最新信息。

2.2 幻觉问题

LLM 本质上是基于概率生成文本，当缺乏确切知识时，会”自信地”编造看似合理但事实上错误的内容——这就是幻觉（Hallucination）。RAG 将生成建立在检索到的事实依据之上，显著降低了幻觉发生的概率。

2.3 领域知识不足

通用 LLM 在垂直领域（医疗、法律、金融等）的知识深度有限，且可能缺乏企业内部的私有数据。RAG 可以接入领域专属知识库，让模型拥有专业能力。

2.4 可追溯性与可信度

纯生成模型的回答如同”黑箱”，用户无法验证信息的来源。RAG 的回答可以追溯到具体的检索文档，提供了可审计的证据链，增强了可信度。

2.5 成本与效率

与通过微调（Fine-tuning）让模型学习新知识相比，RAG 无需重新训练模型，只需更新知识库即可，维护成本远低于微调，且知识更新几乎是实时的。

三、RAG vs 纯生成 vs 纯检索

维度 纯检索（Search） 纯生成（LLM） RAG
知识时效性 ✅ 实时 ❌ 截止于训练数据 ✅ 实时
回答质量 ⚠️ 仅返回原始片段 ✅ 流畅自然 ✅ 流畅且基于事实
幻觉风险 ✅ 无 ❌ 高 ✅ 低
领域适应性 ✅ 取决于知识库 ⚠️ 需微调 ✅ 更新知识库即可
可追溯性 ✅ 有来源 ❌ 无 ✅ 有来源
复杂推理 ❌ 弱 ✅ 强 ✅ 强
维护成本 ⚠️ 中 ❌ 高（需微调） ✅ 低

RAG 的核心优势在于结合了检索的事实基础与生成的推理能力，取两者之长。

四、RAG 核心架构总览

一个完整的 RAG 系统可以分为离线索引和在线查询两个阶段：

离线索引阶段

  1. 文档解析：将各种格式的文档（PDF、HTML、Markdown、表格等）转换为纯文本。
  2. 文本分块（Chunking）：将长文档切分为适当大小的片段，便于检索和送入 LLM 上下文窗口。
  3. 嵌入（Embedding）：使用嵌入模型将文本块映射为高维向量，捕获语义信息。
  4. 向量存储：将嵌入向量存入向量数据库，建立索引以支持高效相似度搜索。

在线查询阶段

  1. 查询嵌入：将用户查询通过同一个嵌入模型转换为向量。
  2. 检索（Retrieval）：在向量数据库中找到与查询向量最相似的文本块。
  3. 重排序（Reranking）：对检索结果进行精细化排序，过滤低相关度内容。
  4. 上下文构建：将重排序后的文本块组织为上下文，与用户查询一起送入 LLM。
  5. 生成（Generation）：LLM 基于上下文和查询生成最终回答。

五、RAG 的演进：从 Naive 到 Modular

随着研究和工程实践的发展，RAG 技术经历了三个主要阶段：

5.1 Naive RAG

最初的 RAG 实现采用最简单的”检索-读取-生成”流程：

优点：实现简单，快速上手
缺点：

  • 检索质量不稳定，可能返回大量无关内容
  • 单一检索策略，无法处理复杂查询
  • 缺乏对检索结果的质量控制
  • 生成时可能出现”Lost in the Middle”问题（模型忽略中间位置的上下文）

5.2 Advanced RAG

在 Naive RAG 基础上引入了多项优化：

关键改进：

  • 查询优化：HyDE（假设性文档嵌入）、Multi-Query（多查询扩展）、Step-back Prompting（后退提问）
  • 混合检索：结合稀疏检索（BM25）与稠密检索（向量相似度），兼顾关键词匹配与语义理解
  • 重排序：使用交叉编码器或 LLM 对检索结果精排
  • 上下文压缩：剔除冗余信息，提取关键片段

5.3 Modular RAG

将 RAG 系统模块化，每个环节可独立替换和优化：

核心特征：

  • 模块化设计：每个组件可独立升级替换
  • 路由机制：根据查询类型选择不同的处理管线
  • 迭代检索：支持多轮检索-生成循环，逐步精化答案
  • 自适应能力：系统可根据评估反馈自动调整策略

三代 RAG 对比

维度 Naive RAG Advanced RAG Modular RAG
检索策略 单一向量检索 混合检索 + 查询优化 可插拔多种策略
后处理 无 重排序 + 压缩 可选多种后处理模块
架构 固定管线 优化管线 可组合模块
检索模式 单次检索 单次/迭代 单次/迭代/自适应
可扩展性 低 中 高
实现复杂度 低 中 高

六、RAG 的典型应用场景

6.1 企业知识问答

将企业内部文档（制度、规范、技术文档等）构建为知识库，员工可通过自然语言提问，RAG 系统检索相关文档并生成准确回答。替代传统的关键词搜索，大幅提升信息获取效率。

6.2 智能客服

基于产品文档、FAQ、历史工单构建知识库，为客户提供精准的自动问答服务。相比传统客服机器人，RAG 能够理解复杂问题并给出基于事实的回答。

6.3 法律/医疗/金融等专业领域

这些领域对准确性要求极高，不容幻觉。RAG 将生成建立在专业文献之上，同时提供来源追溯，确保回答的可信度。

6.4 代码助手

基于代码库构建索引，开发者提问时检索相关代码片段，LLM 结合代码上下文生成回答，实现”理解你的代码库”的编程助手。

6.5 研究与学术辅助

基于论文库构建知识库，帮助研究者快速了解某个领域的关键工作、方法对比和发展脉络。

七、RAG 的局限性

尽管 RAG 显著提升了 LLM 的能力，但它并非万能：

  1. 检索质量瓶颈：如果检索不到相关文档，生成再强也无济于事。”Garbage in, garbage out”。
  2. 上下文窗口限制：LLM 的上下文长度有限，检索结果过多会导致信息截断或稀释。
  3. 多跳推理困难：需要综合多个文档进行推理的问题（如”A 和 B 哪个更早？”），单次检索难以覆盖所有必要信息。
  4. 实时性权衡：知识库的更新频率决定了信息的时效性，高频更新带来索引维护成本。
  5. 复杂查询理解：用户的查询往往模糊、歧义或不完整，简单的嵌入检索难以准确捕捉意图。
  6. 成本与延迟：额外的检索和重排序步骤增加了系统的响应延迟和计算成本。

这些局限性也正是 RAG 研究持续发展的动力——后续文章将深入每个环节的优化策略。
