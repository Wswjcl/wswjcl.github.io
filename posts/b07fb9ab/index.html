RAG 系列（四）：向量存储与索引 — 十亿级检索的幕后

发表于2026-07-09|更新于2026-07-13|技术
|浏览量:

嵌入模型将文本转换为向量后，下一个问题是：如何高效地存储和检索这些向量？当知识库只有几千条记录时，暴力搜索足矣；但当规模扩展到百万甚至十亿级时，就需要专门的向量数据库和近似最近邻（ANN）索引来支撑毫秒级检索。

本文将深入向量存储与索引的核心技术。

一、向量检索的挑战

1.1 维度灾难

给定一个查询向量 $\mathbf{q} \in \mathbb{R}^d$ 和 $N$ 个文档向量 ${\mathbf{v}_1, \dots, \mathbf{v}_N}$，精确最近邻搜索（Exact Nearest Neighbor, ENN）需要计算查询与所有 $N$ 个向量的相似度，时间复杂度为 $O(Nd)$。

当 $N = 10^9$、$d = 1024$ 时，单次查询需要 $10^{12}$ 次浮点运算——即使有 GPU 加速，延迟也远超可接受范围。

1.2 精确 vs 近似

精确搜索保证找到真正的最近邻，但太慢。
近似最近邻搜索（ANN）牺牲少量精度换取数量级的速度提升，在实际中检索召回率可达 95%+，延迟降低 100-1000 倍。

RAG 场景下，ANN 的精度损失完全可接受——因为我们本来就取 Top-K 结果再做重排序，少量遗漏可以通过后续环节弥补。

二、索引结构详解

2.1 Flat 索引（暴力搜索）

最简单的”索引”：不做任何索引，直接遍历所有向量计算相似度。

  • 优点：100% 召回率，实现最简单
  • 缺点：$O(Nd)$ 复杂度，大规模不可用
  • 适用：$N < 10^4$ 的小规模场景

2.2 IVF 索引（倒排文件）

将向量空间划分为 $K$ 个聚类（Voronoi 单元），每个向量归属最近的聚类中心。检索时，只搜索查询最近的 $n_{\text{probe}}$ 个聚类中的向量。

  • 复杂度：构建 $O(NKd)$，检索 $O(\frac{N \cdot n_{\text{probe}}}{K} \cdot d)$
  • 参数权衡：$n_{\text{probe}}$ 越大召回率越高但越慢，通常 $n_{\text{probe}} \ll K$
  • 适用：$10^5 \sim 10^7$ 规模

2.3 HNSW 索引（层级可导航小世界图）

HNSW（Hierarchical Navigable Small World）是目前综合性能最优的 ANN 索引，被 Milvus、Qdrant、Weaviate 等主流向量数据库广泛采用。

核心思想

构建一个多层图结构：

  • 上层：稀疏连接，节点少，用于快速跳跃到查询附近区域
  • 下层：稠密连接，节点多，用于精确定位最近邻
  • 检索时从顶层开始，逐层向下贪心搜索，类似跳表（Skip List）

构建算法

  1. 为新节点随机分配一个层级 $l$（服从几何分布，越高概率越低）
  2. 从顶层入口点开始，逐层向下贪心搜索，找到每层的最近邻
  3. 在第 0 到 $l$ 层，将新节点与其最近的 $M$ 个邻居连接（$M$ 为最大连接数）

层级分配概率：

$$P(l = k) = (1 - m_L) \cdot m_L^k, \quad m_L = \frac{1}{M_0}$$

其中 $M_0$ 是第 0 层的最大连接数。这保证了上层节点稀疏、下层节点稠密。

检索算法

  • GreedySearch：在当前层从入口点出发，每次移动到距离查询更近的邻居，直到局部最优
  • ef：搜索宽度参数，ef 越大召回率越高但越慢

关键参数

参数 含义 典型值
$M$ 非零层的最大连接数 16-64
$M_0$ 第 0 层的最大连接数 2M
$ef_construction$ 构建时的搜索宽度 200-400
$ef_search$ 检索时的搜索宽度 50-200
  • 优点：检索速度极快（$O(\log N)$），召回率高，支持增量插入
  • 缺点：内存占用较大（需存储图结构），构建速度较慢
  • 适用：$10^5 \sim 10^9$ 规模，RAG 中最常用的索引

2.4 量化技术

量化（Quantization）通过压缩向量表示来减少内存占用和加速计算，是大规模向量检索的必备技术。

乘积量化（Product Quantization, PQ）

将 $d$ 维向量切分为 $m$ 个子向量，每个子向量独立量化：

$$\mathbf{v} = [\underbrace{v_1, \dots, v_{d/m}}{\mathbf{v}^{(1)}}, \underbrace{v{d/m+1}, \dots v_{2d/m}}{\mathbf{v}^{(2)}}, \dots, \underbrace{v{d-d/m+1}, \dots, v_d}_{\mathbf{v}^{(m)}}]$$

每个子空间用 K-Means 聚为 $2^8 = 256$ 个聚类中心（码本），向量编码为 $m$ 个 8-bit 码字。

  • 压缩比：$d \times 4$ 字节（float32）→ $m$ 字节（$m$ 个 uint8），通常压缩 32-64 倍
  • 距离计算：使用查找表（Look-up Table）加速，避免解压缩

标量量化（Scalar Quantization, SQ）

将每个浮点数直接量化为 8-bit 整数：

$$v_i^{\text{quant}} = \text{round}\left(\frac{v_i - v_{\min}}{v_{\max} - v_{\min}} \times 255\right)$$

  • 压缩比：4 倍（float32 → uint8）
  • 优点：实现简单，精度损失小
  • 缺点：压缩比不如 PQ

二值量化（Binary Quantization, BQ）

将每个维度量化为 1 bit（正为 1，负为 0），向量编码为位串：

$$v_i^{\text{binary}} = \begin{cases} 1 & \text{if } v_i > 0 \ 0 & \text{if } v_i \leq 0 \end{cases}$$

  • 压缩比：32 倍（float32 → 1 bit）
  • 优点：极致压缩，用汉明距离计算极快
  • 缺点：精度损失大，需要重排序补偿

索引与量化的组合

实际中常将索引结构与量化技术组合使用：

组合 描述 适用规模
IVF + PQ 聚类 + 乘积量化 $10^7 \sim 10^9$
HNSW + SQ 图索引 + 标量量化 $10^6 \sim 10^8$
HNSW + PQ 图索引 + 乘积量化 $10^7 \sim 10^9$
IVF + HNSW + PQ 聚类 + 图 + 量化 $10^9$+

3.2 选型决策

3.3 混合检索的支持

RAG 中混合检索（稀疏 + 稠密）越来越重要。各数据库的支持情况：

  • Milvus：2.4+ 原生支持稀疏向量字段 + 稠密向量字段，混合打分
  • Qdrant：支持稀疏向量（Sparse Vectors），可做混合搜索
  • Weaviate：支持 BM25 + 向量混合（Hybrid Search）
  • pgvector：配合 PostgreSQL 全文搜索实现混合
  • Chroma：不支持原生稀疏检索，需外部实现
  • FAISS：纯向量库，不支持稀疏检索

四、索引构建与维护

4.1 构建策略

策略 描述 适用场景
全量构建 一次性对所有向量建索引 知识库不频繁更新
增量插入 逐条插入向量，实时更新索引 知识库持续更新
批量重建 积攒一批变更后重建索引 更新频率中等，追求索引质量

HNSW 支持增量插入，但大量插入时索引质量不如全量构建（因为后续插入的节点只能连接到已有节点，无法”看到”尚未插入的节点）。

4.2 索引参数调优

HNSW 调优：

参数 调优方向
$M$ 越大 → 召回率越高、内存越大、构建越慢。推荐 16-64
$ef_construction$ 越大 → 构建质量越高、构建越慢。推荐 200-400
$ef_search$ 越大 → 召回率越高、检索越慢。推荐 50-200

经验：先固定 $M=32$，$ef_construction=256$，然后调整 $ef_search$ 在速度和召回率之间权衡。

4.3 分片与分布式

当单机无法容纳所有向量时，需要分片（Sharding）：

  • 按向量 ID 哈希分片：均匀分布，但每个查询需广播到所有分片
  • 按聚类分片：查询只需访问相关分片，减少计算量，但分布可能不均

Milvus 和 Qdrant 都支持分布式部署和自动分片。

五、检索精度与速度的权衡

5.1 Recall vs Latency 曲线

对于任何 ANN 索引，召回率和延迟之间存在 trade-off：

调优目标：在满足延迟要求的前提下最大化召回率。

5.2 两阶段检索

大规模场景下常用两阶段策略：

  1. 粗筛：用量化索引（如 PQ/BQ）快速检索 Top-K’（K’ >> K）候选
  2. 精排：对候选集用原始向量（或高精度量化）重计算精确距离，取 Top-K

这种方式在保持高召回率的同时大幅降低延迟。

六、小结

向量存储与索引是 RAG 系统的”记忆系统”，决定了检索的规模上限和响应速度。本文的核心要点：

  • 精确搜索在大规模下不可行，ANN 是必需的妥协
  • HNSW 是当前综合最优的索引结构，$O(\log N)$ 检索复杂度
  • 量化技术（PQ/SQ/BQ）大幅压缩内存，是十亿级检索的关键
  • 向量数据库选型需考虑规模、混合检索需求、运维成本
  • 两阶段检索（粗筛 + 精排）是大规模场景的最佳实践

Transformer 详解
一、背景与动机在 Transformer 出现之前，序列建模任务（机器翻译、文本生成等）的主流架构是 RNN 及其变体 LSTM / GRU。这些模型虽然在当时取得了不错的效果，但存在几个根本性的局限： 顺序计算瓶颈：RNN 必须逐步处理序列中的每个时间步，无法充分并行化。序列越长，训练越慢。 长距离依赖问题：尽管 LSTM 引入了门控机制来缓解梯度消失，但在实际中捕获超长距离的依赖关系仍然困难。 信息压缩瓶颈：RNN 将整个序列的信息压缩到一个固定维度的隐状态中，信息容易丢失。 2017 年，Vaswani 等人在论文 Attention Is All You Need 中提出了 Transformer 架构，完全摒弃了循环和卷积结构，仅依靠注意力机制（Attention）来建模序列间的依赖关系。这一设计带来了两大优势： 全并行计算：所有位置可以同时计算，极大提升了训练效率。 全局感受野：每个位置可以直接关注序列中任意其他位置，天然支持长距离依赖。 Transformer 的出现彻底改变了 NLP...
